# GridCodec

Maidenhead グリッド伝搬行列用のコンパクトなバイナリコーデック。FT8/WSPR の伝搬データを多数クライアントに効率的に配信するために設計されています。

## 課題

FT8 などの弱信号デジタルモードは大量の伝搬レポートを生成します。4 文字 Maidenhead グリッドは 32,400 格子となり、生の隣接行列は 32,400×32,400、約 **125 MB** のビットマップです。これをテキストや生ビットマップで数百クライアントに配信するのは現実的ではありません。

GridCodec は**階層的次元投影**アルゴリズムで電波伝搬の地理的スパース性を利用し、典型的な実 FT8 活動を **約 2〜20 KB** に圧縮します（1 万倍以上の圧縮）。

## アルゴリズム

### Maidenhead グリッド体系

4 文字 Maidenhead ロケータ（例 `FN31`）は地表の位置を表します。

| 文字   | 意味       | 範囲 | 数   |
| ------ | ---------- | ---- | ---- |
| 1 桁目 | 経度フィールド | A–R  | 18   |
| 2 桁目 | 緯度フィールド | A–R  | 18   |
| 3 桁目 | 経度スクエア   | 0–9  | 10   |
| 4 桁目 | 緯度スクエア   | 0–9  | 10   |

18×18 ＝ **324 フィールド**、各 10×10 ＝ **100 スクエア**に細分され、合計 **32,400 グリッド格子**。

### インデックス方式

```
field_index  = lon_field * 18 + lat_field        [0, 323]
square_index = lon_square * 10 + lat_square      [0, 99]
grid_index   = field_index * 100 + square_index  [0, 32399]
```

### 階層的次元投影

伝搬行列は二値隣接行列で、要素 (i, j) ＝ 1 は「グリッド i から送信された信号がグリッド j で受信された」ことを意味します。

GridCodec はこの行列を直接保存せず、2 層に分解し各層で**次元投影**を適用します。

次元投影は、行/列インデックスが (経度, 緯度) に分解できる 2D ビットマップを 4D テンソルとして扱います：

```
M[src_lon, src_lat, dst_lon, dst_lat]
```

その後、(1) **次元マスクの計算** — 各軸ごとに 1 つのビットマスクで、どの座標値がアクティブな要素に参加しているかを記録。(2) **エントリビットマップの構築** — マスクで定義される範囲内で、どの (lon, lat) の組み合わせが実際に送信元または宛先としてアクティブかをマーク。(3) **内側行列の構築** — アクティブな送信元とアクティブな宛先のみを結ぶ稠密部分行列。

これを 2 段階で適用します：

- **Layer 1（フィールド級）：** 324×324 のフィールド間伝搬行列を圧縮。
- **Layer 2（スクエア級）：** Layer 1 の各アクティブフィールド対について、100×100 のスクエア間部分行列を圧縮。

結果は自己記述バイナリストリームで、すべてのサイズは既にデコードしたデータから導出され、明示的な長さフィールドは不要です。

### 有効な理由

伝搬は地理的にクラスタ化しています。典型的な FT8 バンドでは：

- 324 フィールドのうち 30〜60 のみがアクティブ（アマチュア人口は北米・欧州・日本などに集中）
- アクティブなフィールドは少数の経緯度帯に集中
- 各フィールド対内では、10,000 通りあり得るスクエア間経路の一部のみがアクティブ

次元投影はこの構造を直接捉えます：経度 18 値のうち 5 つ、緯度 18 値のうち 4 つだけがアクティブなら、囲みは 324 エントリから 20 に縮小し、エントリビットマップで真正にアクティブなセルだけに刈り込みます。

## ワイヤフォーマット (v1)

全マルチバイトビットマップは**リトルエンディアン、LSB ファースト**。バイト 0 のビット 0 が最低インデックス要素。

### 全体構造

```
[Header: 2 バイト] [Layer 1] [Layer 2（オプション）]
```

### ヘッダ

| バイト | フィールド | 説明                                                |
| ------ | ---------- | --------------------------------------------------- |
| 0      | version    | `0x01`                                              |
| 1      | flags      | bit 0: `has_layer2`（1 ＝ Layer 1 の後に Layer 2 が続く） |
|        |            | bit 1–7: 予約、0 でなければならない                  |

### Layer 1: フィールド級伝搬

Layer 1 は 324 個の Maidenhead フィールド間の伝搬を符号化します。

**次元マスク**（9 バイト ＝ 4×18 ビット ＝ 72 ビット、パック）：

| ビット範囲 | マスク          | 幅     |
| ---------- | --------------- | ------ |
| 0–17       | src_lon_field   | 18 bit |
| 18–35      | src_lat_field   | 18 bit |
| 36–53      | dst_lon_field   | 18 bit |
| 54–71      | dst_lat_field   | 18 bit |

座標インデックス `i` が任意のアクティブ経路に参加する場合ビット `i` がセットされます。

**送信元フィールドビットマップ：**

- サイズ: `ceil(popcount(src_lon) * popcount(src_lat) / 8)` バイト
- 行優先: 経度が遅く、緯度が速く変化
- ビット (i, j) ＝ 1 は (active_src_lon[i], active_src_lat[j]) のフィールドがアクティブ送信元であることを意味

**宛先フィールドビットマップ：**

- 送信元ビットマップと同じレイアウト、宛先マスクを使用

**内側行列：**

- サイズ: `ceil(n_active_src_fields * n_active_dst_fields / 8)` バイト
- 行優先: 送信元が遅く、宛先が速く変化
- ビット (s, d) ＝ 1 は active_src[s] から active_dst[d] への伝搬が存在することを意味

### Layer 2: スクエア級詳細

`flags & 0x01` のときのみ存在。Layer 1 内側行列の各セットビット（行優先順）ごとに 1 サブブロックが続きます。

各サブブロックは Layer 1 と同じ構造だが、10 ビット次元マスク（lon_square, lat_square）を使用します。

**スクエア次元マスク**（5 バイト ＝ 4×10 ビット ＝ 40 ビット、パック）：

| ビット範囲 | マスク           | 幅     |
| ---------- | ---------------- | ------ |
| 0–9        | src_lon_square   | 10 bit |
| 10–19      | src_lat_square   | 10 bit |
| 20–29      | dst_lon_square   | 10 bit |
| 30–39      | dst_lat_square   | 10 bit |

続いて送信元スクエアビットマップ、宛先スクエアビットマップ、内側行列 — Layer 1 と同一構造で 100×100 スクエア空間で動作。

## 圧縮と性能

生の全行列 ＝ 32,400×32,400 ビット ＝ **125 MB**。

下表は理論的な見積もりと C テストスイートの実測結果をまとめたものです（GCC -O2、x86-64 単一コア）：

| シナリオ                           | フィールド対数 | グリッド経路数 | 符号化サイズ | 圧縮率   | 符号化   | 復号     |
| ---------------------------------- | -------------- | -------------- | ------------ | -------- | -------- | -------- |
| 静かなバンド                       | ~100           | ~1,000         | ~2 KB        | ~64,000× | —        | —        |
| 通常活動（500 経路）\*             | 150            | 500            | 1.7 KB       | ~78,700× | 1.6 ms   | 0.2 ms   |
| 中程度活動（5,000 経路）\*         | 156            | 5,000          | 19.1 KB      | ~6,700×  | 2.7 ms   | 0.4 ms   |
| 繁忙バンド（20,000 経路）\*        | 156            | 20,000         | 105.6 KB     | ~1,200×  | 3.1 ms   | 0.8 ms   |
| 高密度 — 10% ランダム \*           | ~10,500        | ~10.5M         | 12.9 MB      | ~9.7×    | 301 ms   | 131 ms   |
| 高密度 — 11 ホットスポット（ポアソン）\* | 46,955   | ~4.02M         | 4.0 MB       | ~31×     | 801 ms   | 318 ms   |

\* C テストスイートで実測。タイミングのない行はオーダー見積もり。

**11 ホットスポットシナリオ：** 実世界のアマチュア人口中心をモデル化した 11 フィールド（FN 米北東、EM 米南東、DN 米中部、JO 西欧、JN 南欧、IO 英愛、PM 日本、QM 東日本、BN カナダ西、LK インド、OL 中国）、ポアソン分布活動。アクティブグリッドノード 8,131；密度は全行列の 0.38%。

典型的な実 FT8 シナリオ（500–5,000 経路）は符号化 3 ms 未満、復号 0.5 ms 未満で、15 秒 FT8 サイクルに十分収まります。

> **注:** 投影はブロックごとに小さなオーバーヘッド（マスク＋ビットマップ）を付加します。極端な密度（>50% 埋め）では符号化出力が生行列サイズを超えることがあります。アルゴリズムは実電波伝搬に典型的なスパースで地理的にクラスタ化したパターンに最適化されています。

## 実装

GridCodec には参照 C 実装と 4 言語への移植があります。すべて同一ワイヤフォーマット v1 を共有し、相互テストされています（例：非 C 実装はいずれも同一の C 生成テストペイロードを復号し結果を検証）。

### C — 参照実装

#### 単一ヘッダライブラリ

コーデック全体は **`c/gridcodec.h`** にあり、単一ヘッダ C99 ライブラリで、`<stdint.h>` と `<string.h>` 以外の外部依存はありません（デスクトップモードでは `malloc` 用に `<stdlib.h>` を追加）。

**使用パターン：**

```c
#include "gridcodec.h"            /* 宣言のみ */

/* ちょうど 1 つの .c ファイルで： */
#define GRIDCODEC_IMPLEMENTATION
#include "gridcodec.h"            /* 実装を取り込む */
```

全関数は `static` で、各翻訳単位が自分のコピーを持ち、リンク衝突はありません。

#### コンパイル時モード

|                            | デフォルト（デスクトップ/サーバ）   | `GC_EMBEDDED`           |
| -------------------------- | ----------------------------------- | ----------------------- |
| メモリ割り当て             | 動的（`malloc` / `realloc`）        | 静的のみ（約 13 KB）    |
| Layer 2 サポート           | 完全符号化＋復号                    | 復号時にスキップ        |
| `gc_set` / `gc_encode`     | 利用可                              | **コンパイルされない**  |
| `gc_free`                  | 利用可                              | **コンパイルされない**  |
| `gc_from`/`gc_to`（4 文字）| グリッド級結果                      | フィールド級に劣化      |
| `<stdlib.h>`               | 必要                                | **不要**                |
| `popcount`                 | `__builtin_popcount` (GCC/Clang)    | 256 バイトルックアップテーブル |

組込みモードを有効にするには：

```c
#define GC_EMBEDDED
#define GRIDCODEC_IMPLEMENTATION
#include "gridcodec.h"
```

#### API リファレンス

**初期化と解放：**

```c
void gc_init(gc_matrix_t *m);
void gc_free(gc_matrix_t *m);   /* デフォルトモードのみ */
```

**伝搬経路の設定：**

```c
int gc_set(gc_matrix_t *m, const char *from4, const char *to4);
```

2 つの 4 文字グリッド間の経路を記録。冪等 — 重複は内部でビット OR で処理。成功時 `0` を返す。`GC_EMBEDDED` モードでは利用不可。

**符号化と復号：**

```c
int gc_encode(const gc_matrix_t *m, uint8_t *buf, int cap);
int gc_decode(const uint8_t *data, int len, gc_matrix_t *m);
```

`gc_encode` は書き込みバイト数（または `GC_ERR_OVERFLOW`）を返す。`GC_EMBEDDED` モードでは利用不可。

`gc_decode` は消費バイト数を返す。`GC_EMBEDDED` モードでは Layer 2 データは（正しいバイト数を返すために）通過解析されるが破棄される。

**伝搬の問い合わせ：**

```c
int gc_from(const gc_matrix_t *m, const char *grid, int *out, int max_out);
int gc_to(const gc_matrix_t *m, const char *grid, int *out, int max_out);
```

- 2 文字入力（例 `"FN"`）：フィールド索引（0–323）を返す
- 4 文字入力（例 `"FN31"`）：デフォルトモードではグリッド索引（0–32399）、`GC_EMBEDDED` モードではフィールド索引

`gc_from` は与えられた送信元から到達可能な宛先を返す。`gc_to` は与えられた宛先に到達できる送信元を返す。

**索引/名前変換：**

```c
int  gc_field_index(const char *name);     /* "FN"   -> 0..323   */
void gc_field_name(int idx, char out[3]);  /* 0..323 -> "FN\0"   */
int  gc_grid_index(const char *name);      /* "FN31" -> 0..32399 */
void gc_grid_name(int idx, char out[5]);   /* 0..32399 -> "FN31\0" */
int  gc_grid_to_field(int grid_idx);
int  gc_grid_to_square(int grid_idx);
```

名前入力はすべて大文字小文字を区別しません。

**エラーコード：**

| 定数             | 値  | 意味                 |
| ---------------- | --- | -------------------- |
| `GC_ERR_INVALID`  | -1 | 無効なグリッド名     |
| `GC_ERR_OVERFLOW` | -2 | 出力バッファ不足     |
| `GC_ERR_FORMAT`   | -3 | 不正なワイヤフォーマットデータ |
| `GC_ERR_CAPACITY` | -4 | メモリ割り当て失敗   |

#### メモリ使用量

**デフォルトモード：**

| 構成要素               | サイズ                              |
| ---------------------- | ----------------------------------- |
| Layer 1 フィールドビットマップ | 13,122 バイト（固定）        |
| ペアメタデータ         | 38 バイト（ポインタ＋カウンタ）     |
| アクティブフィールド対あたり | 1,254 バイト（送信元＋宛先＋スクエアビット） |

総メモリはアクティブフィールド対の数に比例して増加：

| アクティブ対数 | 総メモリ   |
| -------------- | ---------- |
| 0              | 12.9 KB    |
| 64             | 91.2 KB    |
| 256            | 326.4 KB   |
| 1,024          | 1,266.9 KB |

**組込みモード：** 固定 **13,122 バイト**（12.8 KB）、ヒープ割り当てなし。

必要: C99 コンパイラ（GCC または Clang 推奨）、`make`。


1. **test_gridcodec** — デフォルト（デスクトップ/サーバ）モード：ヘルパー正確性、往復符号化/復号、問い合わせ検証、バッファオーバーフロー、現実的シナリオ（500 / 5,000 / 20,000 経路）、高密度最悪ケース、極端テスト（10% ランダム、11 ホットスポットポアソン）。
2. **test_embedded** — `GC_EMBEDDED` モード：静的メモリサイズ検証、Layer 1 復号、Layer 1+2 復号（Layer 2 は正しくスキップ）、4 文字問い合わせ劣化、符号化 API のコンパイル時不在。

極端テストをスキップ（Valgrind や遅いハードウェアで有用）：

```bash
SKIP_EXTREME=1 ./test_gridcodec
```

メモリリーク検査：

```bash
gcc -g -O0 -std=c99 -o test_dbg test_gridcodec.c
SKIP_EXTREME=1 valgrind --leak-check=full ./test_dbg
```

両モードともリークなしで検証済み（0 エラー、全ヒープブロック解放）。

## 設計判断

**単一ヘッダライブラリ。** 多言語バインディングの統合を簡素化。本リポジトリに Python、MicroPython、JavaScript、Java 実装が同梱；すべて同一ワイヤフォーマットを共有。

**静的関数。** 複数の翻訳単位がヘッダをインクルードしてもシンボル衝突を回避。コンパイラが未使用関数を削除。

**双モードコンパイル。** 組込み機器（ESP32、STM32 等）はヒープゼロ・復号のみの能力が必要。`GC_EMBEDDED` フラグで符号化ロジックと動的メモリを除去し、受信専用局向けの約 13 KB 静的フットプリントを実現。

**線形ペア探索。** `gc_set` と `gc_from`/`gc_to` はフィールド対を線形探索。典型的に 500 対未満ではハッシュテーブルオーバーヘッドより高速。10,000+ 対（極端シナリオ）では測定可能なコスト（約 800 ms 符号化）になるが、バッチサーバ側符号化では許容範囲。

**冪等 `gc_set`。** FT8 復号は同一時間窓内で重複レポートを生成する可能性があるため、`gc_set` は同一ペアで繰り返し呼んでも追加コストがないように設計されています。

**自己記述ワイヤフォーマット。** すべてのフィールドサイズは既にデコードしたデータから導出。長さフィールドが不要で、フォーマットは堅牢：デコーダはストリームを完全に解析するか、エラーコードを返すかのどちらか。
